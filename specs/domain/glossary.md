# Ubiquitous Language — Glossary

> Define domain terms once, use them everywhere.

| Term | Definition | Context | Not to be confused with |
|------|-----------|---------|------------------------|
| **Zone** | Classification of a change by risk: edge (experimental), hybrid (boundary-crossing), core (structural). Determines required artifacts and guardrail rigor. | spec-engine | "Layer" — zones classify changes, layers describe the architecture |
| **Change** | A unit of work tracked in `specs/changes/`. Can be an experiment, improvement, or bugfix. Not every change is a feature. | spec-engine | "Feature" — a change may create a feature, advance one, or have nothing to do with features |
| **Feature** | A product capability tracked across its lifecycle in `specs/domain/features.yaml`. Has a knowledge stage (mystery → heuristic → algorithm). | domain-management | "Change" — features are optional, changes are always tracked |
| **Invariant** | A testable proposition that must ALWAYS hold true within a bounded context. Enforced by fitness functions. The core of evolutionary architecture guardrails. | spec-engine | "Constraint" — invariants are enforced and tested, not just documented |
| **Fitness Function** | An executable test that verifies an invariant holds. Runs via pytest/jest/go test. Can be gated in CI. Neal Ford's term from Evolutionary Architecture. | spec-engine | "Unit test" — fitness functions verify architectural properties, not just code correctness |
| **Knowledge Funnel** | Roger Martin's framework: Mystery → Heuristic → Algorithm. Maps to zones: edge → hybrid → core. Knowledge moves from unknown to codified. | spec-engine | "Pipeline" — the funnel is about knowledge maturity, not delivery stages |
| **Discovery Layer** | The outer layer where knowledge is in the Mystery → Heuristic stage. Design Thinking drives exploration. Specs are hypotheses. The goal is learning. | spec-engine | "Core Engine" — discovery is fast and experimental, core is deliberate and audited |
| **Core Engine** | The inner layer where knowledge is in the Heuristic → Algorithm stage. DDD drives codification. Specs are contracts. The goal is stability. | spec-engine | "Discovery Layer" |
| **Assumption** | A riskiest-first belief that must be true for an experiment to succeed. Tracked through untested → testing → validated/invalidated. Teresa Torres' framework. | spec-engine | "Requirement" — assumptions are hypotheses to test, not commitments to build |
| **Experiment** | A concrete test run against an assumption. Records result, confidence, and decision (continue/pivot/kill/promote-to-core). | spec-engine | "Prototype" — an experiment tests an assumption, a prototype is a tool used in an experiment |
| **Kill Criteria** | Specific, measurable conditions for abandoning an experiment. Prevents over-investing in ideas that don't work. | spec-engine | "Acceptance criteria" — kill criteria define when to stop, not when to ship |
| **Promote to Core** | The decision to move validated discovery knowledge into the core domain. A heuristic becomes an invariant with a fitness function. | spec-engine | "Ship" — promoting to core means codifying the rule, not deploying code |
| **Upstream** | A sibling repository whose domain entities and invariants are visible to this project. Referenced in `evospec.yaml → upstreams[]`. | domain-management | "Dependency" — upstreams share domain knowledge, not code dependencies |
| **Published Contract** | A repo's `specs/domain/` files treated as its public domain API. Other repos reference these via upstreams. | domain-management | "API contract" — published contracts are domain knowledge, not HTTP endpoints |
| **Canonical Workflow** | A platform-agnostic YAML file defining a workflow (e.g., discover, capture). The generator reads these and emits Windsurf, Claude, and Cursor files. | agent-integration | "Slash command" — the workflow is the definition, the slash command is one platform's interface to it |
| **Capture** | Retroactive formalization of experimental work into an implementation spec + domain artifacts. The AI scans code to generate docs. | agent-integration | "Document" — capture is an automated process, not manual documentation |
| **Implementation Spec** | An as-built blueprint documenting what was actually implemented: architecture, APIs, state, config, reproduction instructions. Created by `/evospec.tasks` (deliberate) or `/evospec.capture` (experimental). | spec-engine | "Design doc" — implementation specs describe what IS, not what SHOULD BE |
| **Domain Contract** | A formal specification of entities, invariants, state machines, events, and authorization rules within a bounded context. Required for core zone. | spec-engine | "Discovery spec" — domain contracts codify rules, discovery specs explore hypotheses |
| **ADR** | Architecture Decision Record. Documents why a decision was made, alternatives considered, and consequences. Nygard's format. | spec-engine | "Spec" — ADRs record decisions, specs guide implementation |
| **Entity Registry** | The canonical list of domain entities in `specs/domain/entities.yaml`. Machine-readable. Populated by `evospec reverse db` or manually. Exposed via MCP `evospec://entities`. | domain-management | "Schema" — the registry describes domain concepts, not database tables |
| **Bounded Context** | A DDD strategic design concept. A linguistic boundary where a domain model has a specific meaning. Registered in `specs/domain/contexts.yaml`. | domain-management | "Module" — bounded contexts are semantic boundaries, not code boundaries |
